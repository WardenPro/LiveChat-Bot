### US-002: Define module boundaries and compatibility map
Status: in_progress
Depends on: US-001

Description:
As a maintainer, I want a target module map with compatibility constraints so that code reorganization remains safe and consistent.

Acceptance Criteria:
- [ ] Create a documented module map covering loaders, components, services, repositories, and shared utilities for the full codebase.
- [ ] For each moved/renamed internal file, provide compatibility re-export wrappers where internal consumers need a phased transition.
- [ ] Example: existing internal import path continues to resolve via compatibility wrapper during migration phase.
- [ ] Negative case: any proposed module move that would alter an external entrypoint is rejected and logged as out of scope.
- [ ] Boundary rules include prohibited cross-layer dependencies and approved dependency directions.
