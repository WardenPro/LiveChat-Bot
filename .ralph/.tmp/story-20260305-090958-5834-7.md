### US-007: Standardize safe error handling and logging
Status: in_progress
Depends on: US-006

Description:
As an operator, I want consistent error handling and safe logs so that failures are diagnosable without leaking sensitive details.

Acceptance Criteria:
- [ ] Create typed error categories and centralized mapping to HTTP/socket/command error outputs while preserving current observable status/body behavior.
- [ ] Ensure sensitive values (tokens, secrets, raw credentials) are never logged.
- [ ] Example: known operational error logs structured context and returns the same user-visible error response contract.
- [ ] Negative case: unexpected thrown error does not crash long-running services and still returns baseline-equivalent safe response.
- [ ] Error-handling paths are covered by characterization scenarios in touched modules.
