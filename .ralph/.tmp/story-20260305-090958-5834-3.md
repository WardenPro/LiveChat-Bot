### US-003: Refactor REST route loading into domain modules
Status: in_progress
Depends on: US-002

Description:
As a maintainer, I want REST route registration and handlers organized by domain so that HTTP behavior is easier to maintain without changing contracts.

Acceptance Criteria:
- [ ] Reorganize REST loader internals into clearly separated domain registrars while preserving route prefixes, methods, and handler behavior.
- [ ] Maintain existing response payload keys and status codes for admin, overlay, and ingest routes.
- [ ] Example: GET /overlay/config returns the same fields and status code before and after refactor.
- [ ] Negative case: unsupported route paths continue to return the same not-found behavior as baseline.
- [ ] Characterization checks for REST routes pass with no contract diff.
